# PRODUCT_SENSE.md

> 产品判断原则。在功能设计存在分歧时，用此文档做决策。

## 核心原则

### 1. 减少用户决策负担

用户面对的选择越少，完成任务越快。默认值应该是 90% 场景下的最优解，而不是中立值。

### 2. 错误应该难以发生

设计应引导用户走向正确路径，而非依赖他们"看文档"。表单限制、状态约束、操作确认都是防御设计。

### 3. 响应必须及时

- 操作反馈 < 100ms（视觉响应）
- 数据加载 < 1s（用骨架屏）
- 超过 3s 的操作必须有进度提示

### 4. 失败必须可恢复

删除需要确认，危险操作需要可撤销或有警告。用户不应因误操作丢失数据。

## 功能取舍标准

在决策一个功能是否值得做时：

| 问题 | 判断依据 |
|------|--------|
| 影响多少用户？ | < 10% 的用户场景，推迟或不做 |
| 是否可用其他方式满足？ | 优先复用已有能力 |
| 维护成本如何？ | 高复杂度功能需明确 owner |
| 是否与核心路径冲突？ | 不能让主流程变复杂 |

## UI 决策原则

- **一致性优于创新**：新交互模式带来学习成本，优先复用 Ant Design 已有模式
- **内容优于装饰**：信息密度 > 视觉效果
- **渐进披露**：高级设置默认折叠，避免干扰主流程
